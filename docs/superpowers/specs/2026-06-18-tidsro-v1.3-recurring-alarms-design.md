# Tidsro v1.3 — Recurring (weekday) alarms (Slice 3) — Design Specification

- **Status:** Approved (brainstormed) — ready for planning
- **Date:** 2026-06-18
- **Author:** Malin Fossum
- **Type:** Desktop application feature design spec (v1.3)
- **Builds on:** v1.1 (clock-time alarms) and v1.2 (design/interaction polish); the original [Tidsro design spec](2026-06-03-tidsro-design.md) and the [v1.1 spec](2026-06-16-tidsro-v1.1-clock-time-alarms-design.md)

---

## 1. Context & scope

v1.1 shipped one-shot clock-time alarms; v1.2 was a design/interaction polish pass. This is **Slice 3: recurring (weekday) alarms** — an alarm that repeats at a fixed time on a chosen set of weekdays, living in the same agenda (renamed **"Schedule"**). It is the feature that turns Tidsro into a set-once-and-forget daily focus tool.

v1.1 deliberately paved this road: `TriggerType.Recurring` already exists; the agenda already sorts armed alarms by next fire time; and the one-shot firing model was built so recurring could extend it rather than replace it.

**In scope**
- A recurring alarm: a 24-hour time + a set of weekdays (Mon–Sun), repeating weekly until deleted.
- Friendly presets (Daily / Weekdays / Weekends) as quick-fills over the same day-set, plus a custom day picker.
- Recurring and one-shot alarms in **one unified "Schedule" list**, sorted by next occurrence, the soonest pinned to the top.
- Fire once per occurrence, then roll forward; survive sleep / relaunch within the 5-minute grace; quiet missed note beyond it.
- Reuse of the completion card (Snooze +5 / Dismiss) and the modal Edit dialog + undo-guarded delete.

**Out of scope (parked or later)**
- A separate "Recurring" section, or a day-filtered agenda — **parked**; revisit after the unified list ships.
- Intervals ("every 2 days"), monthly, specific calendar dates, end dates.
- Custom user sound files, cloud sync, code signing / auto-update, light theme — unchanged non-goals.

---

## 2. Decisions locked during brainstorming

1. **Recurrence model = weekday-set.** A time + any subset of Mon–Sun, weekly, forever until deleted. Presets are combinations of the same day-set, not a separate model. *(Q1 = A)*
2. **One unified "Schedule" list.** Each alarm (one-shot or recurring) is a single row with a cadence tag, sorted by next occurrence, the soonest always pinned to the top with the "next" dot. The two-section and day-filtered layouts are **parked** as possible future enhancements. *(Q2 = A)*
3. **Rename the zone "Your day" → "Schedule".**
4. **Missed-while-away = calm, same as one-shots.** Within the 5-minute grace it still fires; beyond it, one quiet note, then it arms the next occurrence. A long absence collapses to one note + at most one catch-up fire. *(Q3 = A)*
5. **Editor = a Repeat dropdown** (Once / Daily / Weekdays / Weekends / Custom…) in **both** the add-editor and the modal Edit dialog; Custom reveals seven M–S toggles; default **Once** = today's exact one-shot behaviour. *(Q4 = A)*
6. **Runtime = extend `TimerItem`** with one nullable `RecurringDays` field and keep all alarms in the single `_alarms` list; recurring is a branch in `Tick`, not a parallel system. *(Fork a)*
7. **Dedup via roll-forward, not a separate ledger.** A recurring alarm advances its persisted `EndsAt` to the next occurrence the instant it fires, so the persisted next-occurrence is always the next *unhandled* one — the same durability as a one-shot's persist-on-fire, with no new `LastFired` field.

## 3. Feature spec

### 3.1 The recurring alarm

A recurring alarm is a time-of-day + a `Weekdays` set + an optional label + a per-alarm sound (silent by default). It repeats at that time on each selected weekday, every week, until deleted. With **zero days selected the editor produces a one-shot** (today's behaviour); recurring begins at one or more days.

### 3.2 The "Schedule" zone (renamed from "Your day")

The add-editor gains a Repeat control; the agenda gains a cadence tag. Otherwise the v1.2 layout, focus model, and empty state are unchanged.

**Add-editor** — one new control after Sound:
- **Repeat** — a dropdown: Once / Daily / Weekdays / Weekends / Custom…. Default **Once**. Picking **Custom** reveals a row of seven day toggles (Mon–Sun); the presets simply set the toggles. Time / Label / Sound / Add are unchanged; **Enter** still adds.
- **On Add:** if the resolved day-set is empty → the existing one-shot path (`ArmClockAlarm`); otherwise `ArmRecurringAlarm`. The alarm is armed, persisted, and inserted in sorted position; the editor resets to Once.

**Agenda** (one unified list):
- Every alarm is one row, sorted by next occurrence (`EndsAt`), then label, then `Id`. The soonest row carries the gold **"next"** dot. As alarms fire and roll forward, the list re-sorts so the next alarm is always on top — which is why no day-filtered view is needed.
- Row contents: the time (bold anchor), the label (or "No label"), a **cadence tag**, a **sound indicator** (icon + text), edit + delete.
- **Cadence tag** (icon/text, never colour alone): one-shots show `once` or `tomorrow`; recurring show `Daily`, `Weekdays`, `Weekends`, or an explicit list like `Mon Wed Fri`.
- Edit / delete unchanged: edit opens the modal dialog; delete is undo-guarded (disarm at once, re-arm on undo, commit on timeout/quit).

### 3.3 Editing a recurring alarm (modal dialog)

The pencil opens the existing `EditAlarmWindow`, now carrying the Repeat control too, pre-filled from the row (time, label, sound, days). **Save** validates the time, applies the edit in place (same `Id`, re-armed to the new next occurrence, re-persisted, re-sorted) and closes. **Cancel / Esc** discards. Changing the day-set or the time recomputes the next occurrence.

### 3.4 Firing, roll-forward, sleep, and launch

- The existing tick processes recurring alarms alongside one-shots and countdowns; the alarm loop branches on `TriggerType`.
- For a recurring alarm where `now >= EndsAt`:
  - Compute `prev = RecurrenceRules.MostRecentOccurrence(now, time, days)` — the latest matching weekday at HH:MM ≤ `now`.
  - **Within the 5-minute grace** (`now - prev <= Grace`) → fire: play the sound and raise the completion card (via a transient occurrence item, §3.5). **Beyond the grace** → expire quietly into the missed-while-away note (§3.6).
  - **Either way**, advance `EndsAt = RecurrenceRules.NextOccurrence(now, time, days)` (strictly future) and `SaveData()`.
- **On launch:** each loaded recurring alarm is armed by the same rule — a persisted next-occurrence due within grace fires once, an older one expires into the note, otherwise it simply arms for its next occurrence. The arm pass is wrapped so a bad record can never crash launch.
- **A long absence** (several missed occurrences) collapses to one outcome: at most one catch-up fire (only if `now` is within grace of the most recent occurrence) and a single quiet note — never a stack — then it arms the next future weekday. Advancing `EndsAt` past the handled occurrence is the dedup, so neither a 250 ms tick nor a quick relaunch re-fires it.

### 3.5 Completion card (reused, with one guard)

A fired recurring occurrence raises the same bottom-right card as a one-shot, with **Snooze +5 / Dismiss** (`ShowRestart = false`).

**Guard:** the card's Dismiss calls `SchedulerService.Cancel(item)`. For a one-shot that is harmless (it left the armed set at fire time); for a *live* recurring alarm it would **delete the whole schedule**. So a recurring fire raises the card with a **transient occurrence item** — a lightweight copy carrying the label and sound (typed so the card shows the alarm actions) that is **not** in `_alarms`. Snooze and Dismiss act on that copy; the live recurring alarm stays armed and now shows its next date. Snooze +5 spawns the usual transient 5-minute countdown; Dismiss just closes the card.

### 3.6 Missed-while-away note

Unchanged from v1.1: alarms that expired beyond the grace surface **one quiet, dismissible line** (no sound, no card), announced once via the UIA notifier. A recurring alarm's note shows its time and label like any other; because its time-of-day is constant, the HH:MM shown is always the occurrence's time.

## 4. Data model

**New — `Weekdays` (a `[Flags]` enum):**

```
[Flags]
enum Weekdays { None = 0, Mon = 1, Tue = 2, Wed = 4, Thu = 8, Fri = 16, Sat = 32, Sun = 64 }
```

Presets are combinations: `Weekdays` = Mon|Tue|Wed|Thu|Fri, `Weekends` = Sat|Sun, `Daily` = all seven. Serialises as one integer.

**Runtime** — a recurring alarm reuses `TimerItem`:
- `TriggerType = Recurring`.
- `EndsAt` = the next occurrence (so the existing `now >= EndsAt` tick fires it unchanged).
- **New nullable field `Weekdays? RecurringDays`** (null for countdowns and one-shots) so the tick can roll forward.
- `Label`, `Sound`, `State` as today; `State` stays `Running` across fires (it never permanently leaves the armed set).

**Persisted** — new DTO `RecurringAlarmRecord`:

```
RecurringAlarmRecord {
  Guid        Id;
  int         Hour;        // 0–23
  int         Minute;      // 0–59
  Weekdays    Days;
  string?     Label;
  SoundChoice Sound;
  DateTime    NextFireAt;  // local; the next occurrence — the durable dedup marker
}
```

**Persistence document** — `TidsroData` widens to schema 3:

```
TidsroData {
  int                        SchemaVersion;     // 3
  AppSettings                Settings;
  List<AlarmRecord>          Alarms;            // one-shots (unchanged)
  List<RecurringAlarmRecord> RecurringAlarms;   // new
}
```

- `CurrentSchema` 2 → 3; `Defaults()` adds an empty recurring list.
- **Back-compat is automatic:** a v2 (or v1.0) file has no `RecurringAlarms` key → the list loads empty; the existing tolerant load and v1.0→v2 fallback are untouched. The next save writes it forward as schema 3.
- **`Sanitized()` extends to recurring records:** drop any with `Days == None`, `Hour` outside 0–23, `Minute` outside 0–59, a non-representable `NextFireAt` (the existing `IsRepresentable` range), an undefined `Sound`, or a duplicate `Id` (keep first); labels trimmed and length-capped as today. No polymorphic / `$type` handling.
- The launch arm pass stays wrapped so any residual bad record quarantines → defaults rather than crashing startup.
- **Save triggers** unchanged in spirit: the whole document is written atomically on add, edit, delete-commit, and whenever the tick fires / expires / advances an alarm or a snooze changes the running set.

**New — `RecurrenceRules` (mirrors `ClockTimeRules`):**
- reuses `ClockTimeRules.TryParse` for the time;
- `NextOccurrence(DateTimeOffset now, int hour, int minute, Weekdays days)` → the soonest matching weekday at HH:MM strictly after `now`, in local wall-clock;
- `MostRecentOccurrence(DateTimeOffset now, int hour, int minute, Weekdays days)` → the latest matching weekday at HH:MM at or before `now`;
- a small helper to render a `Weekdays` set as a cadence label (Daily / Weekdays / Weekends / "Mon Wed Fri").

## 6. Key behaviours & edge cases

- **Fire once per occurrence (durable):** advancing `EndsAt` to the next occurrence + an immediate `SaveData()` is the dedup — no double-fire across a 250 ms tick, a sleep gap, or a quick relaunch.
- **Multi-miss:** a week away → one note + at most one catch-up fire, then armed for the next weekday.
- **Dismiss / Snooze on a recurring fire:** act on a transient occurrence copy; the live recurring alarm stays armed (Snooze also spawns a transient 5-minute countdown).
- **Edit / Delete:** modal edit recomputes the next occurrence; undo re-arms preserving days + next occurrence; an uncommitted delete commits on quit.
- **DST / time zones:** occurrences are built in local wall-clock, so a 07:00 alarm fires at 07:00 across a DST shift. Spring-forward's missing hour (02:00–03:00) is the only oddity — an alarm set in that hour fires at the system's resolved wall-clock; acceptable for a desk timer and consistent with the one-shot stance.
- **Empty day-set in the editor = a one-shot** (the Once default), so the existing one-shot path is never disturbed.

## 7. Accessibility

- Inherits v1.1 / v1.2: full keyboard operation, visible focus, logical tab order, WCAG-AA dark contrast, reduced-motion respect, the completion-card keyboard path + UIA announcement.
- The **Repeat dropdown** carries `AutomationProperties.Name`; the seven **day toggles** each carry a full day name ("Monday") and a pressed / unpressed state, not colour alone; presets are reachable and announced.
- The **cadence tag is text** ("Weekdays"), folded into the row's accessible name (e.g. *"Alarm at 07:00, Morning, weekdays, chime, next"*), so a screen-reader user hears the cadence.
- Add / edit / delete each announce via the UIA notifier as today (e.g. *"Alarm added for 07:00, weekdays"*).
- Focus management unchanged: edit moves focus into the dialog and returns to the row on close; delete moves focus to the undo control.

---

## 8. Privacy & security

- Unchanged posture: fully local, no telemetry, no network. Recurring alarms persist only to `%AppData%\Tidsro\data.json`.
- Labels stay out of all logs; loaded JSON stays untrusted and range-validated (§4). The `.corrupt` quarantine (which may carry labels) is still cleared on the next good save.

## 9. Testing

Backend is **TDD** (Model / Services); the view is **manual acceptance** — as in every slice.

Unit tests (`dotnet test`):
- `RecurrenceRules`: next and most-recent occurrence across single days, presets, and full custom sets; week wrap (e.g. Sunday → next Monday); the same-day boundary (time still ahead vs already passed); a DST-boundary day. An empty set is treated as one-shot by the caller, not by the rules.
- `Weekdays`: preset compositions; cadence-label rendering (Daily / Weekdays / Weekends / "Mon Wed Fri").
- `SchedulerService`: arm computes the right next occurrence; a recurring alarm fires at its occurrence and **advances instead of being removed**; within-grace catch-up fires once; beyond grace expires once; a daily alarm fired at 07:00 does **not** re-fire on a tick 30 s later, nor on a relaunch 1 min later (persisted advanced `EndsAt`); recurring and one-shot / countdown lists don't cross-contaminate.
- **Durable single-fire & delete lifecycle:** a fired recurring occurrence persists its advanced `EndsAt` so a reload neither re-fires nor falsely reports "missed"; a deleted recurring alarm fires neither during nor after the undo window; undo before its next occurrence re-arms it with its days intact.
- `PersistenceService` / `TidsroData`: schema-3 round-trip; **v2 and v1.0 back-compat** (load with an empty recurring list, settings + one-shots preserved); sanitisation drops a day-less, out-of-range, extreme-`NextFireAt`, undefined-sound, or duplicate-`Id` recurring record; the launch arm pass never throws on a crafted file.
- ViewModel logic where it warrants: add resolves preset / custom → `Weekdays`; the one-shot vs recurring branch; the cadence tag; edit applies the new day-set.

Manual acceptance (with a screen reader for the a11y items): add a recurring alarm via each preset and via Custom; the unified sort with the next alarm pinned on top; firing + Snooze / Dismiss leaving the alarm armed; a missed-while-away note after sleep; the Repeat dropdown + day-toggle legibility and keyboard path; the cadence tag; edit changing the day-set; delete + undo.
