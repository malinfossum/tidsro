# Tidsro v1.1 — Clock-time Alarms (Slice 2) — Design Specification

- **Status:** Approved (stress-tested) — ready for planning
- **Date:** 2026-06-16
- **Author:** Malin Fossum
- **Type:** Desktop application feature design spec (v1.1)
- **Builds on:** v1.0.0 (countdowns, Slice 1) and the original [Tidsro design spec](2026-06-03-tidsro-design.md)

> **Revision 2026-06-16 (post stress-test):** Folded in nine findings from the spec stress-test — durable fire-once (persist on fire / expire / snooze, §3.3, §4); delete disarms immediately and re-arms on undo, committing on quit (§3.1, §6); the delete-undo is keyboard-focusable, announced, and non-brief (§3.1, §7); loaded alarms are range-validated so the launch arm pass can never throw (§4); edit-mode and add / edit / delete focus + announcements are pinned (§3.2, §7); snooze's session-only durability and the card debounce are named (§3.4, §6); and the quarantine file's label exposure is handled (§8).

---

## 1. Context & scope

v1.0.0 shipped Slice 1 — countdown timers. This is **Slice 2: clock-time alarms** — one-shot "fire at HH:MM" alarms. It turns the main window's "Your day" zone (an empty-state placeholder in v1.0) into a live agenda, and adds the app's first persisted *user* data (alarms, alongside the settings already saved).

The original design spec (§5.1, §5.2, §6) already planned this slice; this document refines it with the decisions made during v1.1 brainstorming and pins the implementation against the v1.0 codebase.

**In scope**
- Create a one-shot clock-time alarm: a 24-hour time, an optional label, a per-alarm sound (silent by default).
- A live "Your day" agenda: alarms sorted by fire time, with in-place edit and undo-guarded delete.
- Firing that survives sleep and app-reopen within a 5-minute grace; quiet expiry beyond it.
- Alarm persistence across restarts.
- Reuse of the completion card with alarm-appropriate actions (Snooze +5 / Dismiss).
- Ride-along polish: the light-on-dark ComboBox dropdown fix (the sound picker uses it).

**Out of scope (later slices)**
- Recurring / weekday alarms → Slice 3 (the data model and firing here are designed not to block it).
- Custom user sound files, cloud sync, code signing / auto-update, light theme — unchanged from the v1.0 non-goals.

---

## 2. Decisions locked during brainstorming

1. **Headline = clock-time alarms only.** Recurring stays in Slice 3.
2. **Fired-alarm card = Snooze +5 / Dismiss.** "Restart" (re-run a duration) is hidden for alarms — it has no meaning for a one-shot.
3. **Robustness = tick-gap + 5-minute grace + fire-once-via-state.** No Win32 power-resume listener and no `LastFired` ledger — a one-shot fires, flips to `Fired`, and is removed, which is its own dedup. (The heavier machinery is genuinely needed only for recurring, so it waits for Slice 3.)
4. **Row tag = sound, not cadence.** Every v1.1 alarm is one-shot, so a "once" tag is noise. Each row shows a silent-vs-sound indicator (icon **and** text). A "daily / once" tag returns with recurring.
5. **Persist an absolute `FireAt`** (local date + time), not just a time-of-day — a one-shot needs to know *which day* it fires, which makes reload, grace, and expiry unambiguous.
6. **Missed-while-away = a quiet note.** Alarms missed beyond the grace window surface one calm, dismissible line — no sound, no card — rather than vanishing silently.
7. **Edit = in-place, reusing the inline editor.** The pencil loads an alarm back into the top editor in an "editing" mode; Save updates it in place.

## 3. Feature spec

### 3.1 The "Your day" zone

Replaces the v1.0 empty-state. Two parts: an inline add/edit editor, then the agenda.

**Inline editor** (top of the zone):
- **Time** — a 24-hour `HH:MM` field, the focus target when you enter the zone (and the keyboard entry point). Parsed and validated by a new `ClockTimeRules.TryParse` (mirroring `CountdownRules`); invalid input shows a calm inline message and nothing is added.
- **Label** — optional free text.
- **Sound** — a dropdown, default **Silent**, seeded from the global default sound (as the countdown picker is). **▶ preview** as in v1.0. This dropdown carries the ride-along ComboBox dark-dropdown fix.
- **Add** — primary (gold) button. On success the alarm is armed, persisted, and inserted into the agenda in sorted position; the editor clears.
- If the chosen time has already passed today, the alarm rolls to **tomorrow** at that time, and its agenda row carries a subtle "tomorrow" tag so the day is never ambiguous.

**Agenda** (below the editor):
- Alarms sorted by `FireAt`, then label, then `Id` (stable tie-break).
- Each row: the time (the bold anchor), the label (or a muted "No label"), a "tomorrow" tag when the alarm isn't today, a **sound indicator** (icon + text — e.g. a music glyph + "chime", or a muted glyph + "silent"), and **edit** + **delete** buttons.
- The **next alarm to fire** is marked with a small gold dot.
- **Empty state** until the first alarm: "Nothing scheduled yet — add an alarm."
- **Delete is undo-guarded** (spec §5.2): the row leaves immediately and the alarm is **disarmed at once** (pulled from `_alarms` so it cannot fire mid-undo), then a dismissible undo appears. The undo is a real **keyboard-focusable** control; **focus moves to it** on delete, and it is **announced** via the UIA notifier. It does **not** auto-commit on a brief flash — its window has a comfortable floor (≈ 8–10 s), or it commits only on explicit dismiss. If the window lapses, the delete commits (dropped from the scheduler and from disk); **undo re-arms** the alarm, re-checking it against the grace rule if its `FireAt` passed in the meantime. An uncommitted delete commits on app quit. No confirm dialog.

### 3.2 Editing an alarm (in-place)

- The pencil on a row loads that alarm's time / label / sound into the inline editor, which enters **editing mode**: the edited row is highlighted, **Add** becomes **Save**, a **Cancel** appears, and **focus moves to the time field**.
- **Save** validates as on add, updates the existing alarm in place (re-armed to the new `FireAt`, re-persisted, re-sorted), and returns the editor to add mode. **Save / Cancel return focus to the affected row.**
- **Cancel** / **Esc** exits editing with no change.
- **Edit-during-fire:** if the alarm being edited would fire mid-edit, firing is **suppressed** for that moment (the in-progress edit wins, per spec §8). On Save it is re-evaluated against the new `FireAt`; on Cancel the original `FireAt` applies and is re-checked against the grace rule — so opening edit, dawdling past the fire time, then cancelling lets the original alarm **expire into the missed-while-away note** (§3.5) rather than firing late. This is intended, not accidental.

### 3.3 Firing, sleep, and launch

- One scheduler tick (the existing 1-second `DispatcherTimer`) checks armed alarms alongside running countdowns.
- **On tick / on resume from sleep** — for each armed alarm where `now >= FireAt`:
  - within the **5-minute grace** → fire: `State → Fired`, raise the existing `Fired` event (→ sound + corner card), then remove the alarm (one-shot complete). The `Running → Fired` flip is the fire-once guard, so a sleep-induced tick gap cannot double-fire.
  - beyond the grace → **expire quietly**: remove it, no sound, no card, and record it for the missed-while-away note (§3.5).
- **On launch:** alarms are loaded and armed by the same rule — future `FireAt` → arm; `FireAt` ≤ 5 minutes past → fire once; older → expired (and noted).
- A one-shot is **never rolled forward** on reload — `FireAt` is an absolute moment, so "missed" means missed, not "do it tomorrow".
- **Persist on change:** whenever the tick fires or expires an alarm (and on snooze), the armed set has changed, so `SaveData()` runs and the one-shot leaves disk at once. This is what makes the fire-once guard **durable** — without it, a process kill seconds after a fire would leave the alarm on disk and let it re-fire (within grace) or show as falsely "missed" on the next launch (§4, §6).

### 3.4 Completion card (reused)

- A fired alarm raises the same corner card as a countdown, via the existing `OnTimerFired` path — bottom-right, topmost-but-non-activating, persists until dismissed, with the keyboard path and UIA announcement all inherited.
- `PopupViewModel` gains `ShowRestart`, set **false** for `TriggerType.ClockTime`. The card then shows **Snooze +5** and **Dismiss**.
- **Snooze +5** reuses `SchedulerService.Snooze(item, 5 min)` → a fresh 5-minute countdown (it appears in the Quick-timers running list like any countdown, and fires its own card in 5 minutes). Because the snooze *is* a countdown, it is **session-only**: a snoozed alarm does **not** survive an app restart (consistent with countdown semantics, §6). The original alarm was removed at fire time, so nothing lingers as an alarm.
- **Dismiss** closes the card; the alarm was already removed at fire time.
- **Snooze / Dismiss inherit the existing completion-card debounce** (spec §8) — a fast double-click performs the action once (no double snooze).

### 3.5 Missed-while-away note

- Alarms that expired beyond the grace (during sleep or while the app was closed) are surfaced as **one quiet, dismissible line** — e.g. *"Missed while away: Lunch · 14:00"* — listing the expired alarm(s). No sound, no card, no action beyond dismiss.
- Announced once via the existing UIA notifier so a screen-reader user isn't left unaware.

## 4. Data model

**Runtime** — a clock-time alarm reuses `TimerItem`:
- `TriggerType = ClockTime`.
- `EndsAt` = the armed fire moment (`FireAt` as `DateTimeOffset`) — so the existing tick (`now >= EndsAt`) fires it unchanged.
- `Label`, `Sound`, `State` as today.
- Armed alarms live in a **new, separate `SchedulerService._alarms` list**, kept apart from `_running` countdowns so they never render as timer rows and have their own lifecycle.

**Persisted** — a new DTO, distinct from the runtime item:

```
AlarmRecord {
  Guid          Id;
  DateTime      FireAt;   // local; absolute date + time
  string?       Label;
  SoundChoice   Sound;
}
```

Recurring (Slice 3) will add its own representation (time-of-day + days-of-week) without disturbing this.

**Persistence document** — `PersistenceService` widens from "settings only" to a root document:

```
TidsroData {
  int                 SchemaVersion;   // 2
  AppSettings         Settings;
  List<AlarmRecord>   Alarms;
}
```

- **Atomic save / tolerant load** unchanged in spirit (tmp + `File.Replace`; quarantine-to-`.corrupt` + defaults on a bad file; never fail to launch).
- **Back-compat:** a v1.0 `data.json` is a bare `AppSettings`. If the document parse yields no `Settings`, fall back to reading the file as a bare `AppSettings` and adopt it (with an empty alarm list), then save forward as `TidsroData`. No one loses their startup / sound / window prefs.
- **Untrusted load extended to alarms:** a `TidsroData.Sanitized()` pass calls the existing `AppSettings.Sanitized()` and drops any `AlarmRecord` with an undefined `SoundChoice`, a duplicate `Id` (keep first), or a `FireAt` that is `default`, unparseable, **or outside a sane representable range** — extreme `DateTime` values (e.g. `DateTime.MaxValue`) are rejected here, not just `default`, because they would otherwise throw on the `DateTime → DateTimeOffset` conversion when arming. Labels are trimmed and length-capped on load. No polymorphic / `$type` handling, as before.
- **The launch arm pass can never throw:** arming loaded alarms is wrapped so any residual bad value quarantines → defaults rather than crashing startup, honouring the v1.0 promise that a bad data file must never stop the app launching.
- **Save triggers:** the document is written (atomically, whole) on add, edit, delete-commit, and whenever the tick fires / expires / snoozes an alarm — so disk always reflects the live armed set. Settings still save on settings changes, through the same `SaveData()`.

## 5. How it maps onto the v1.0 code

- `Models/TriggerType.cs` — already `Countdown | ClockTime | Recurring`; no change.
- `Models/TimerItem.cs` — no new fields for the runtime alarm (it reuses `EndsAt`).
- `Models/AlarmRecord.cs` — **new** persisted DTO.
- `Models/ClockTimeRules.cs` — **new**: parse/validate "HH:MM" (0–23 : 0–59), compute today-or-tomorrow `FireAt`.
- `Models/AppSettings.cs` — unchanged; becomes the `Settings` member of `TidsroData`.
- `Services/PersistenceService.cs` — load/save `TidsroData`; add the back-compat fallback, alarm sanitisation (range/dup-Id), and the `.corrupt` cleanup (§8).
- `Services/SchedulerService.cs` — add `_alarms`, `ArmClockAlarm(...)`, `Alarms` (read-only), `RemoveAlarm(...)`; extend `Tick()` to process alarms with the grace/expiry rule; make `Snooze`/`Cancel` correct for an item in either list.
- `ViewModels/PopupViewModel.cs` — add `ShowRestart` (false for `ClockTime`).
- `ViewModels/MainViewModel.cs` — real `IsDayEmpty`; the `Alarms` agenda; editor state (`AlarmTimeInput`, `AlarmLabel`, `AlarmSound`, `AlarmError`, editing-mode flag/target); `AddOrSaveAlarmCommand`, `BeginEditAlarm`, `CancelEditAlarm`, `DeleteAlarm` (disarm + undo); `RefreshAll` reconciles the agenda (drop fired/expired, keep sorted).
- `ViewModels/AlarmItemViewModel.cs` — **new**: one agenda row (time text, label-or-"No label", tomorrow tag, sound indicator, is-next flag, edit/delete).
- `Views/MainWindow.xaml` — render the editor + agenda in the "Your day" zone; ComboBox dark-dropdown fix on the sound picker.
- `App.xaml.cs` — on startup, load `TidsroData` and arm alarms from records (the arm pass is wrapped so it can't crash launch); a single `SaveData()` (settings + alarms) replaces the settings-only save callback and runs on add / edit / delete-commit **and after the tick fires / expires an alarm**; pending deletes commit on quit; the missed-while-away note is surfaced after the launch arming pass.

## 6. Key behaviours & edge cases

- **Fire at most once (durable):** `Running → Fired` + removal *and* an immediate `SaveData()` (§3.3) — no double-fire across a sleep tick-gap or a process kill. (No `LastFired` needed for one-shot.)
- **Past time on add / edit:** rolls to tomorrow; the row shows "tomorrow".
- **Missed while asleep / closed:** ≤ 5 min late → fire once; older → expire + note.
- **Edit during fire:** firing suppressed; re-evaluated on Save / Cancel (a dawdle-then-cancel past the fire time expires the original into the missed note, §3.2).
- **Delete then undo:** delete disarms the alarm immediately (it cannot fire during the undo window); undo restores and re-arms it, re-checking the grace rule if its `FireAt` passed meanwhile; an uncommitted delete commits on quit. Only a committed delete removes it from disk.
- **Snooze:** spawns a 5-minute countdown (transient, session-only) — a snoozed alarm does not survive an app restart (§3.4). The original alarm is already gone (fired), so nothing reappears as a missed note.
- **Duplicate times / labels:** allowed; the stable sort keeps their order fixed.
- **DST / time zones:** all local; `FireAt` is a local `DateTime`. A one-shot set across a DST boundary fires at the wall-clock time shown — acceptable for one-shot; recurring will revisit.
- **Large tick gap with several alarms due:** each armed alarm is evaluated independently against the grace rule on the same tick; multiple due alarms stack cards bottom-right (existing behaviour).

## 7. Accessibility

- Inherits v1.0: full keyboard operation, visible focus, logical tab order, WCAG-AA dark contrast, reduced-motion respect, the completion-card keyboard path + UIA announcement.
- The editor fields carry `AutomationProperties.Name` (Time, Label, Sound). Inline validation errors are associated with the time field and announced via the UIA notifier.
- Each agenda row exposes an accessible name like *"Alarm at 14:00, Meeting prep, chime"*; edit / delete buttons carry per-alarm accessible names.
- The **sound state is carried by icon + text**, never colour alone; the "tomorrow" and "next" cues likewise have text / structural equivalents (a screen-reader name says "tomorrow", not just a dot).
- **Focus management:** entering edit moves focus to the time field; Save / Cancel returns focus to the edited row; deleting moves focus to the undo control (never to nothing). The completion-card focus-return to the prior app is inherited.
- **State changes are announced:** add, edit, and delete-commit each raise a UIA notification (e.g. "Alarm added for 14:00"), matching the undo and missed-while-away announcements — so a screen-reader user isn't left guessing whether an action took.

---

## 8. Privacy & security

- Unchanged posture: fully local, no telemetry, no network. Alarms persist only to `%AppData%\Tidsro\data.json`.
- **Labels remain out of all logs** (they are routinely personal). `AlarmRecord` is plain data; loaded JSON stays untrusted and range-validated (§4).
- **Quarantine files may carry labels:** a tolerant-load failure copies `data.json` to `data.json.corrupt` for recovery, which now includes alarm labels. This is accepted for its recovery value, but the stale `.corrupt` file is **deleted once a subsequent good save succeeds**, so personal labels don't linger indefinitely after a recovery.

## 9. Testing

Backend is **TDD** (Model / Services); the view is **manual acceptance** — as in v1.0.

Unit tests (`dotnet test`):
- `ClockTimeRules`: valid "HH:MM"; bounds (`24:00`, `12:60`, negatives, junk); today-vs-tomorrow `FireAt`.
- `SchedulerService`: arm computes the correct `FireAt`; tick fires at/after `FireAt`; **grace boundary** (fires within 5 min late, expires past it); one-shot removed after fire — a second tick does not re-fire; snooze produces a 5-minute countdown; alarms and countdowns don't cross-contaminate their lists.
- **Durable fire-once & delete lifecycle:** a fired one-shot is persisted out at fire time, so a reload neither re-fires it nor reports it "missed"; a deleted alarm fires neither during nor after the undo window, while undo before its `FireAt` re-arms it.
- **Launch arm-from-records:** future arms; ≤ 5 min past fires once; older expires (and is reported for the note).
- `PersistenceService`: `TidsroData` round-trip; **v1.0 back-compat** (a bare-`AppSettings` file loads with settings preserved + empty alarms); sanitisation drops a bad / extreme `FireAt`, an undefined sound, and duplicate `Id`s, and the launch arm pass never throws on a crafted file; corrupt file → defaults + quarantine, and a stale `.corrupt` is cleared after a good save (existing test extended).
- ViewModel logic where it warrants: sorted insert + `IsDayEmpty`; delete-undo commit vs cancel; editing-mode Save updates in place.

Manual acceptance (with a screen reader for the a11y items): add / edit / delete an alarm, the tomorrow roll, firing + Snooze / Dismiss, a missed-while-away note after sleep, the ComboBox dropdown legibility, and focus landing correctly on edit and delete.
